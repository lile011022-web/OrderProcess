# AGENTS.md

本文件是「球星卡采购接单对账系统」的长期开发规范。每次开始开发、修改、重构或排查问题前，必须先阅读本文件，并在完成后按本文件要求说明改动与验证方式。

## 1. 项目定位

本项目是一个面向球星卡采购、接单、包裹入库、费用核算、付款确认、异常处理和对账管理的业务系统。

第一阶段只做前端原型：

- 不接真实后端。
- 不调用真实接口。
- 使用集中维护的 mock data 模拟业务数据。
- 页面必须可运行、可交互、可维护。
- 不允许只有空页面、占位页面或静态图片式页面。

## 2. 技术栈

固定使用以下技术栈：

- React
- Vite
- TypeScript
- Tailwind CSS

新增依赖前必须确认是否必要。优先使用项目已有能力和轻量实现，避免引入大型、不必要的库。

## 3. 核心开发原则

必须遵守企业级前端开发规范：

- 组件复用：通用 UI、业务卡片、表格、状态标签、筛选器、弹窗等应抽成可复用组件。
- TypeScript 类型清晰：业务对象、角色、状态、费用明细、mock 数据都必须有明确类型。
- 目录结构清晰：按职责拆分页面、组件、路由、权限、费用计算、mock 数据、类型定义。
- 权限逻辑集中管理：不要在各个页面里散落硬编码角色判断。
- 费用计算逻辑集中管理：仓库操作费、照片费、汇率换算、成本确认等规则必须统一封装。
- mock 数据集中管理：不要在页面组件里临时写大段业务数据。
- 不要把所有代码写在一个文件里。
- 不要写空页面。
- 页面必须具备真实业务含义和基本交互能力。
- 代码应方便后续替换为真实后端接口。

## 5. 角色与权限

系统必须支持四种角色：

- `admin`：管理员
- `buyer`：买手
- `warehouse`：仓库
- `customer`：客户

不同角色登录后必须看到不同菜单和不同可操作范围。客户页面必须和管理员页面分开，客户只维护自己的商品资料和仓库地址提交，管理员负责全局维护与审核。

权限规则必须集中维护，例如：

- 菜单权限集中配置。
- 路由访问权限集中配置。
- 操作按钮权限集中配置。
- 页面内业务动作根据统一权限函数判断是否显示或可点击。

禁止在多个页面里重复写散落的角色判断，例如大量出现：

```ts
user.role === 'admin'
```

应优先封装为统一函数或配置，例如：

```ts
canAccessRoute(role, routeKey)
canUseAction(role, actionKey)
getMenusByRole(role)
```

## 6. 登录与路由规则

必须支持登录态逻辑：

- 未登录访问后台页面时，必须跳转到 `/login`。
- 已登录用户访问后台时，根据角色展示对应菜单。
- 登录页应允许在前端原型阶段选择或模拟四种角色登录。
- 登出后应清理登录态，并回到 `/login`。

第一阶段可以使用 localStorage 或 React state 模拟登录态，但必须集中封装，不要在页面里重复读写。

## 7. 角色菜单要求

菜单应按角色区分，示例方向如下：

管理员 `admin` 可看到：

- 总览
- 采购订单
- 包裹管理
- 仓库费用
- 付款与确认
- 异常包裹
- 对账报表
- 用户与权限

买手 `buyer` 可看到：

- 我的采购订单
- 我的包裹
- 付款记录
- 异常处理
- 对账查看

仓库 `warehouse` 可看到：

- 待处理包裹
- 入库确认
- 仓库照片
- 异常登记
- 费用确认

客户 `customer` 可看到：

- 我的商品资料
- 我的仓库地址
- 提交审核状态

客户页面不得复用管理员审核页面。客户只能查看和维护自己提交的数据，管理员在基础资料中查看全量数据并执行审核、启用或停用。

实际菜单可以根据业务继续细化，但必须满足不同角色看到不同菜单。

## 8. 运单号查询规则

系统中的运单号必须支持点击跳转到对应官网查询：

- UPS
- FedEx
- USPS

运单跳转逻辑必须集中维护，不要在多个组件里拼接链接。

推荐封装：

```ts
getTrackingUrl(carrier, trackingNumber)
```

当 carrier 未知或运单号为空时，应给出清晰的禁用态或提示，避免跳转到错误地址。

## 9. 仓库操作费规则

仓库操作费必须集中计算，禁止在页面组件里散落计算公式。

当前固定规则：

- 包裹操作费：5 美元 / 包裹
- 仓库照片费：0.5 美元 / 张
- 固定汇率：1 美元 = 6.8 人民币

推荐统一封装：

```ts
calculateWarehouseFee({
  packageCount,
  photoCount,
})
```

计算结果应至少包含：

- 包裹操作费美元金额
- 照片费美元金额
- 合计美元金额
- 合计人民币金额
- 使用的汇率

所有页面展示费用时，应使用同一个费用计算模块和格式化函数。

## 10. 付款与入库成本规则

系统必须清晰区分以下两个概念：

- 已付待确认金额
- 实际入库成本

业务规则：

- 买手或管理员付款后，该金额只能进入“已付待确认金额”。
- 仓库确认实际收到后，才能转为“实际入库成本”。
- 未经仓库确认，不允许直接计入实际入库成本。
- 页面上必须让用户能看出金额处于待确认还是已确认状态。

相关状态、金额字段和展示文案必须命名清晰，避免把已付款误认为已入库成本。

## 11. 异常包裹规则

异常包裹必须支持记录和后续处理。

异常可能产生：

- 退款
- 下次抵扣

异常处理逻辑应能表达：

- 异常原因
- 责任方或处理人
- 涉及金额
- 处理方式：退款或下次抵扣
- 当前处理状态
- 备注或凭证信息

异常金额不应直接混入正常入库成本，应在对账或费用汇总中清晰区分。

## 12. Mock 数据规范

第一阶段必须使用 mock data，但 mock 数据必须集中管理。

要求：

- mock 用户、订单、包裹、付款、异常、对账数据应放在 `src/mock/` 或等价集中目录。
- mock 数据必须使用 TypeScript 类型约束。
- 页面不应直接定义大量 mock 数组。
- mock 数据应覆盖四种角色的常见业务场景。
- mock 数据应覆盖正常、待确认、异常、已退款、下次抵扣等状态。

后续接真实后端时，应尽量只替换数据访问层，不大改页面组件。

## 13. 页面质量要求

每个页面都必须是可用原型，而不是空壳。

页面至少应具备：

- 清晰标题和业务说明性状态。
- 有真实感的 mock 数据。
- 基础筛选、搜索、切换、查看详情、状态变化等交互之一。
- 对不同角色的可见数据或可操作项有差异。
- 空数据状态、禁用状态或异常状态有合理展示。

后台类页面应注重效率和可扫描性，避免做成营销落地页风格。

## 14. 组件规范

组件应按职责拆分：

- 布局组件：导航、侧边栏、顶部栏、页面容器。
- 通用组件：按钮、状态标签、表格、弹窗、筛选器、金额显示。
- 业务组件：订单摘要、包裹卡片、费用明细、付款确认、异常处理。
- 页面组件：组合业务流程，不承载过多底层逻辑。

组件文件应保持可读，不要把多个复杂组件堆在一个文件里。

## 15. TypeScript 规范

必须为核心业务对象定义类型：

- 用户与角色
- 菜单与权限
- 采购订单
- 包裹
- 承运商与运单
- 仓库费用
- 付款记录
- 入库成本
- 异常处理
- 对账记录

避免滥用 `any`。如果临时无法确定类型，应优先定义最小可用类型，并在后续迭代中完善。

## 16. 状态与业务文案规范

业务状态必须使用统一枚举或联合类型维护。

示例：

```ts
type PaymentStatus = 'unpaid' | 'paid_pending_confirm' | 'confirmed_received'
type ExceptionResolution = 'refund' | 'next_credit'
```

页面展示文案应通过统一映射维护，避免同一个状态在不同页面显示成不同中文。

## 17. 开发前检查

每次开发前必须：

1. 阅读 `AGENTS.md`。
2. 查看当前项目结构。
3. 明确本次修改涉及的页面、组件、类型、mock、权限、费用或路由。
4. 优先沿用现有代码风格和目录结构。
5. 避免无关重构。

## 19. 运行与验证

常规运行方式：

```bash
npm install
npm run dev
```

常规验证方向：

- `/login` 是否可访问。
- 未登录访问后台页面是否跳转 `/login`。
- 四种角色是否能登录或模拟登录。
- 四种角色登录后菜单是否不同。
- 运单号是否能跳转到 UPS / FedEx / USPS 官网查询。
- 仓库费用是否按 5 美元 / 包裹、0.5 美元 / 张照片、汇率 6.8 正确计算。
- 付款后是否先进入“已付待确认金额”。
- 仓库确认后是否才进入“实际入库成本”。
- 异常包裹是否能表达退款或下次抵扣。

## 20. 长期维护要求

本项目应长期保持：

- 业务规则集中。
- 页面真实可操作。
- 类型清晰。
- 组件可复用。
- mock 可替换。
- 权限可扩展。
- 费用计算可审计。
- 对账流程表达准确。

当业务规则变化时，应优先更新集中规则模块和本文件，再同步调整页面与 mock 数据。
